\begin{abstract}
AI assistants can decompose multi-step workflows, but they do not natively speak industrial protocols such as Modbus, MQTT/Sparkplug B, or OPC UA. This paper presents \system, a suite of Model Context Protocol (MCP) adapters that expose industrial operations as schema-discoverable AI tools while preserving protocol-specific connectivity and safety controls. The system uses a common response envelope and a mock-first workflow so adapter behavior can be exercised locally before connecting to plant equipment. We evaluate the current flagship adapters for Modbus, MQTT/Sparkplug B, and OPC UA with deterministic mock environments and fixed tool workflows corresponding to 14 representative assistant tasks. Across 140 normal runs and 200 tool calls, all benchmarked tasks completed successfully; recovery experiments after forced endpoint restarts also recovered in all 30 trials. The results position \system\ as a reproducible prototype for AI-assisted industrial operations, while also making clear that the evidence is limited to mock-based adapter behavior rather than production deployment.
\end{abstract}

\section{Introduction}
\label{sec:introduction}

Consider a simple operator request: check a temperature excursion on an OPC UA namespace, confirm a related Modbus register block, and publish a Sparkplug B status update for downstream systems. Each subtask is routine in isolation, but the overall workflow crosses incompatible operational technology (OT) protocols whose interfaces were not designed for modern AI assistants. This protocol gap remains a practical barrier to AI-assisted industrial automation even as Industry 4.0 programs push tighter coupling between information technology and plant-floor systems \cite{xu2018industry4,wollschlaeger2017future}.

Large language model (LLM) assistants work best with typed, discoverable tool interfaces rather than binary protocol frames or vendor SDKs. MCP provides a standard way for assistants to list tools, inspect schemas, and invoke operations through a common call interface \cite{anthropic2024mcp}. MCP alone, however, does not solve OT connectivity, write-guarding, or the need to validate behavior safely before touching real equipment.

This paper reframes \system\ as a bounded systems contribution: a prototype MCP-to-OT adapter architecture with a mock-first evaluation workflow. The paper does not claim a complete industrial platform, a new LLM reasoning method, or production deployment evidence. Instead, it focuses on the flagship adapters for Modbus, MQTT/Sparkplug B, and OPC UA, while retaining seven additional protocol modules as ecosystem breadth.

The manuscript makes three contributions:
\begin{enumerate}
    \item A practical MCP-to-OT adapter architecture that standardizes tool discovery and response handling across heterogeneous industrial protocols.
    \item A mock-first methodology for validating adapter behavior, write guards, and restart recovery without requiring physical devices.
    \item A reproducible prototype evaluation on Modbus, MQTT/Sparkplug B, and OPC UA using deterministic workloads, latency instrumentation, and fault-injection trials.
\end{enumerate}

\section{Design Goals and Architecture}
\label{sec:architecture}

\textbf{Heterogeneity.} Industrial protocols expose different interaction models: register maps for Modbus, broker topics for MQTT, and browsable information models for OPC UA \cite{cavalieri2013opcua,mishra2020mqtt}. The adapter layer must translate these differences into tool schemas that remain understandable to an AI client.

\textbf{Schema discoverability.} Each adapter registers MCP tools so a client can enumerate available operations and call them without bespoke protocol code. This keeps the assistant-facing interface uniform even when the underlying transport or addressing model differs.

\textbf{Write safety.} The adapters expose explicit write operations, but writes remain bounded by per-server configuration such as disable flags, address validation, and structured error returns. These controls matter because an AI-facing API must represent both successful writes and intended denials in a machine-readable way.

\textbf{Mockability.} Every flagship adapter is paired with a local mock so the full tool path can be exercised offline. This enables reproducible evaluation, debugging, and regression checks before any real controller or fieldbus is involved.

\begin{figure}[t]
    \centering
    \includegraphics[width=0.9\columnwidth]{figures/fig1_architecture.png}
    \caption{\system\ exposes industrial protocol clients as MCP servers so an AI assistant can invoke them through a common tool interface while targeting either mocks or real devices.}
    \label{fig:architecture}
\end{figure}

Figure \ref{fig:architecture} shows the core pattern. An MCP-compatible assistant issues tool calls to an \system\ adapter. The adapter handles protocol translation, connection management, and any safety guards, then talks to either a mock endpoint or plant equipment. All benchmarked adapters return the same envelope,
\texttt{\{success, data, error, meta\}}, where \texttt{meta} carries endpoint details and latency measurements. This does not erase protocol differences; instead, it gives a common wrapper around those differences so higher-level orchestration can remain consistent.

In the current prototype, this envelope is also the benchmark boundary. The adapters record wall-clock latency, retry attempts, and endpoint-specific metadata directly in the tool response. That makes cross-family comparison possible without pretending that a broker publish, a Modbus register read, and an OPC UA browse are identical operations.

\label{sec:implementation}

\textbf{Modbus.} The Modbus adapter centers on register and coil operations, connection health checks, and guarded writes. In the evaluated workflow it handled register reads, write-and-readback verification, and explicit write denial when writes were disabled. The recovery benchmark also exercised reconnect behavior after a forced Modbus mock restart.

\textbf{MQTT/Sparkplug B.} The MQTT adapter exposes broker health, topic subscriptions, generic publishes, and Sparkplug B device-data publishes. Its MCP responses normalize broker connectivity and publish status into the common response envelope while preserving topic-level semantics. In practice this lets one assistant workflow mix ordinary MQTT control messages with Sparkplug-oriented telemetry updates.

\textbf{OPC UA.} The OPC UA adapter exposes node browsing, node reads, writes, and multi-node access against a local mock plant. Method invocation exists in the prototype but was excluded from the benchmark because the current mock method path is not yet stable enough for repeatable evaluation. The measured workload therefore focuses on repeatable assistant-facing operations instead of the full prototype surface.

\section{Experimental Setup}
\label{sec:setup}

The evaluation targets adapter behavior rather than end-to-end LLM planning. To keep the workload reproducible, we fixed a set of representative assistant intents and executed them as deterministic MCP tool workflows against local mocks. This isolates connection handling, tool schemas, response normalization, and safety behavior from prompt variability \cite{yao2022react}.

        \bottomrule
    \end{tabularx}
\end{table}

Table \ref{tab:tasks} summarizes the workload. The normal benchmark matrix contains 14 tasks: four Modbus tasks, four MQTT/Sparkplug B tasks, four OPC UA tasks, and two cross-protocol tasks. The cross-protocol tasks exercise a multi-adapter state snapshot and a coordinated control sequence spanning all three flagship stacks. Each task was repeated 10 times, yielding 140 normal runs and 200 tool calls overall.

The workload includes one guarded-write scenario in which the Modbus adapter is configured read-only and is expected to return a structured denial rather than perform the write. Task success and tool-call success are therefore judged against a task-specific oracle, not simply against \texttt{success=true}. A structured write denial counts as correct behavior for that task.

The benchmark harness starts the local mocks, opens MCP stdio sessions for each adapter, executes the workload, and records JSON results from the shared response envelope. We measured task success, tool-call success, median latency, p95 latency, and retry counts from those records. We also injected one transient outage per flagship stack by stopping the mock or broker, confirming an operation failure, restarting the endpoint, and re-running a healthy operation. Each recovery scenario was repeated 10 times for a total of 30 restart trials.

Table \ref{tab:results} shows clean success rates across the full benchmark: 140 of 140 normal task executions succeeded, and all 200 tool calls matched their expected outcomes. No retries were required in the normal suite. Modbus and MQTT/Sparkplug B remained in the low single-digit millisecond range for both median and p95 latency. Cross-protocol workflows also completed reliably, showing that the common tool surface is usable for simple multi-adapter orchestration.

Task-level behavior is also informative. Modbus medians ranged from 1.19 ms for a simple ping to 2.85 ms for write-and-readback validation. MQTT/Sparkplug B stayed tightly grouped between 1.72 and 1.80 ms across broker inspection, subscription, and publish operations. Cross-protocol medians were 6.43 ms and 6.74 ms for the control and snapshot tasks, respectively.

OPC UA showed the largest tail latency. The baseline node read, browse, and write-readback tasks remained modest at 2.24 ms, 7.29 ms, and 4.50 ms median latency, but the variable-enumeration task reached 108.94 ms median and drove the family p95 to 114.86 ms. This is consistent with the information-model breadth of OPC UA rather than a generalized slowdown across all OPC UA operations.

    \includegraphics[width=\columnwidth]{generated/flagship_recovery_latency.png}
    \caption{Median end-to-end recovery latency after forced endpoint restarts. All three flagship stacks recovered in 10 of 10 trials and detected the outage in 10 of 10 trials.}

Figure \ref{fig:recovery} summarizes the restart experiments. All three flagship stacks recovered in every trial after a forced outage. The measured recovery times are dominated by mock or broker restart time plus reconnection, not by the steady-state tool latency shown in Table \ref{tab:results}.

The failure modes also differed by stack. MQTT typically failed fast because the adapter could detect a disconnected broker immediately, while OPC UA recovery was slower because the failed read path waited on a longer timeout before the restarted server became reachable again. This suggests that future hardening work should tune timeout and reconnect policy per protocol rather than enforce a single global default.

\textbf{Limitations.} The evidence is intentionally narrow. This is a mock-only study, not a production deployment evaluation, not a cybersecurity validation, and not evidence that AI agents can safely perform closed-loop autonomous control on plant equipment. The benchmark also omits human baselines and excludes OPC UA method calls because the current prototype mock does not yet support them reliably enough for repeatable measurement.

\section{Related Work}
\label{sec:related}

OT/IT convergence literature has long identified heterogeneous field protocols as a core integration problem for Industry 4.0 systems \cite{xu2018industry4,wollschlaeger2017future}. OPC UA and MQTT are widely used normalization layers in modern industrial software, but each solves only part of the interoperability problem because brownfield plants still depend on multiple legacy and vendor-specific stacks \cite{cavalieri2013opcua,mishra2020mqtt}.

Recent LLM-agent work emphasizes reasoning plus external tool use rather than monolithic natural-language interfaces \cite{yao2022react}. MCP contributes a practical, schema-driven interface for tool discovery and invocation \cite{anthropic2024mcp}. Our contribution is not a new tool-use protocol; it is the adapter pattern, shared safety conventions, and benchmark evidence needed to map that protocol into industrial operations.

The mock-first workflow is adjacent to simulation and digital-twin practices used for industrial software validation \cite{fuller2020digital}. In contrast to full digital-twin objectives, the focus here is narrower: verify adapter behavior, structured failures, and restart recovery before introducing real equipment.

\section{Conclusion}
\label{sec:conclusion}

\system\ is best understood as a prototype MCP-to-OT adapter architecture rather than a complete industrial platform. The current evidence shows that the Modbus, MQTT/Sparkplug B, and OPC UA adapters can execute deterministic mock workloads with consistent response semantics, guarded-write behavior, and repeatable restart recovery. The broader repository provides ten protocol modules, but only three are evaluated in this paper.

The next steps are straightforward: validate the same workflows on real hardware, harden the security and approval path around write-capable tools, and expand the benchmark beyond mock connectivity into operator-in-the-loop scenarios.
